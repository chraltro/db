SELECT c.customer_id, c.name, COUNT(o.order_id) AS order_count, SUM(o.total_amount) AS lifetime_value FROM silver.customers c LEFT JOIN silver.orders o ON c.customer_id = o.customer_id
📊 Query Plan EXPLAIN ANALYZE
├── PROJECTION [4 columns] │ customer_id, name, order_count, lifetime_value │ ├── HASH_GROUP_BY [groups: 2, aggregates: 2] │ Groups: c.customer_id, c.name │ Aggregates: count(o.order_id), sum(o.total_amount) │ Estimated Cardinality: 847 │ ├── HASH_JOIN [LEFT, condition: c.customer_id = o.customer_id] │ Estimated Cardinality: 12,450 │ │ ├─ SEQ_SCAN silver.customers [847 rows] │ │ Columns: customer_id, name │ │ Rows: 847 | Time: 0.3ms │ │ │ └─ SEQ_SCAN silver.orders [12,450 rows] │ Columns: customer_id, order_id, total_amount │ Rows: 12,450 | Time: 1.2ms
💡
Why this matters: Users can now understand query performance without executing the full query. This is especially useful for large tables where running a bad query could lock the warehouse. The EXPLAIN view maps to DuckDB's EXPLAIN ANALYZE or EXPLAIN statements.
